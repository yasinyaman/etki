## Repository status

**Faz 0–4 are implemented** (walking skeleton → data backbone → decision brain → HITL/audit/UI → pilot & productization). Two paths coexist:

- **Online (triage):** `POST /triage` → `CaseFile`, served from a **persisted real index** (`.etki/index.json`). No Joern/JVM/LLM at request time.
- **Offline (indexing):** `python -m etki.indexing` builds that index from real adapters — FileSystem documents → heuristic scope extraction (INCLUDED/EXCLUDED) → Baseline; **live Joern** code graph; scope↔code mapping. The CLI **delegates to `context.index_project`** (2026-07-11) — same path as the UI's reindex: doc_root composite, multi-repo merged graph, DB-baseline reconcile (its old raw-connectors copy fed a project the FAKE corpus and could overwrite approved CRs). `IndexingEngine.build` **skips unparseable documents** with a warning instead of failing the whole project.

Faz 2 added the **decision brain** (all deterministic / auditable): full decision tree in `engine/triage.py` (maintenance → EXCLUDED → code+text two-evidence → limit/quota → effort-pool), three-point/PERT range estimation (`engine/estimation.py`), rule-based request understanding (`engine/understanding.py`), risk escalation, and an **MCP server** (`etki/mcp_server.py`) + **LLM agent** (`etki/agent.py`) that expose the index to an LLM agent. The LLM is **off by default** (deterministic gate); it is a seam for request-splitting enrichment, CR-draft prose, the agent, and **semantic/cross-lingual triage matching** — when the deterministic scope match is weak (e.g. a Turkish request against an English spec/codebase), `TriageEngine._llm_match` asks the LLM to map the request to the right scope clause + impacted modules (fallback only, so triage stays fast/free on strong matches; logged as an "LLM destekli eşleştirme" assumption). **Two providers** behind the same `LLMClient` port (config, not code — `registry.build_llm_client`): **Anthropic Claude API** (`adapters/llm_anthropic.py`, official `anthropic` SDK, default model `claude-opus-4-8`) enabled via `ETKI_LLM_PROVIDER=anthropic` + `ANTHROPIC_API_KEY`; or **OpenAI-compatible** (`adapters/llm_openai.py`, Ollama/vLLM) via `ETKI_LLM_BASE_URL`. No key/endpoint → falls back to heuristic/deterministic.

Faz 3 added the **human-in-the-loop layer** (all deterministic): SQLAlchemy persistence behind a `CaseFileRepository` port (SQLite for tests/CI, Postgres for live — connection string only), an approval workflow (`hitl/service.py`: APPROVE/REJECT/CONVERT_TO_CR; CR approval → **living baseline version+1**), a full **audit trail** so any decision is reconstructable, **RBAC** (`api/security.py`; approval requires the `pmo` role), KPI/over-reliance dashboard (`kpi.py`), and an **HTMX server-rendered UI** (`api/web.py` + `api/templates/`; case-file review with evidence + approve buttons, KPI scorecard — **UI fully redesigned 2026-06**, see **UI** below). KVKK/security notes in `docs/KVKK.md`.

Faz 4 added **pilot & productization**: a **shadow-mode pilot** (`etki/pilot/`, `python -m etki.pilot`) that runs the system over a labeled pilot set and reports decision agreement, effort-in-range, per-decision precision/recall, and **confidence calibration** — without affecting the live workflow; **config-driven decision thresholds** (`in_scope_threshold`/`gray_threshold` in Settings + `TriageEngine`) with a **calibration** loop (`pilot/calibration.py`) that suggests threshold changes from pilot/override mismatches (human-applied); a **composite document adapter** (`adapters/composite_document.py`) proving pluggability (config-only swap, core untouched — `tests/integration/test_composite.py`); and **containerization** (`Dockerfile` + `docker-compose.yml` app+Postgres, `config/connectors.docker.yaml` uses the `ast` engine so the container needs no JVM) + `docs/RUNBOOK.md`.

**Multi-project** is supported: `config/projects.yaml` lists projects (each with its own `id`, `contract_id`, connectors); each project gets its own index (`.etki/index-{id}.json`) + baseline + `TriageEngine`, all keyed by `project_id` in `api/context.py` (`AppContext.engines`). ## What Etki is

A human-in-the-loop **PMO decision-support system**. For each incoming client request it answers: *is this in-scope, out-of-scope, or a Change Request (CR); will it blow the effort budget; what does it cost?* The defensible IP is the **fusion of three sources** to produce a live, evidence-backed recommendation — no off-the-shelf product does this combination:

1. **Contract scope** — clauses extracted into structured `ScopeItem`s (including explicitly *excluded* ones).
2. **Code knowledge graph** — modules, dependencies, complexity, churn — for impact analysis and effort.
3. **Historical effort** — real time logged on past tickets, used by analogy.

## Architecture invariants (the non-obvious, load-bearing decisions)

These are the design rules that span many files and are easy to violate. Preserve them.

- **Hexagonal ports & adapters — the core is vendor-agnostic.** The decision engine, indexer, and monitor never reference a concrete vendor (no `Jira`/`GitHub`/`SharePoint` strings in core). They talk only to three abstract ports:
  - `WorkItemProvider` — tickets + spent effort, normalized to `WorkItem.effort_seconds`.
  - `CodeRepositoryProvider` — repo access, churn/commit history, incremental diff.
  - `DocumentSourceProvider` — document listing + content fetch.
  Adapters (Jira, GitHub, GitLab, FileSystem, SharePoint…) implement a port and **normalize** vendor differences away. Which adapter is active is **configuration, never code**. Adding a vendor = writing one new adapter; core layers must not change. This is the single biggest productization multiplier — do not let vendor specifics leak into core.
  - Adapters declare **capabilities** (`supports_webhooks`, `supports_realtime`, `supports_effort_tracking`, `supports_incremental_diff`); the system **degrades gracefully** (e.g. polling when no webhooks, full re-index when no incremental diff).

- **Two phases at different cadences.** *Indexing* is offline/periodic (parse code → knowledge graph; parse contracts → baseline). *Triage* is online and must answer in seconds by querying the pre-built indexes. Keep heavy work in indexing; keep triage a fast lookup.

- **EXCLUDED scope is first-class.** `ScopeItem.polarity` is `INCLUDED | EXCLUDED`. A request matching an EXCLUDED item is the highest-confidence out-of-scope call ("explicitly excluded, clause X"). Never collapse polarity into a boolean "in-scope" flag.

- **Baseline is versioned and live.** Each approved CR appends a new `ScopeItem` and bumps the baseline version; later requests are judged against the *expanded* scope. The baseline is the single source of truth.

- **Two-evidence rule in the decision engine.** A decision rests on both (a) text similarity (request vs. contract) and (b) code reality (does the request touch already-scoped code). Agreement → high confidence; conflict → **gray area → escalate to PMO**. The decision tree order is: maintenance-scope → negative/EXCLUDED match → code+text match → limit/quota → effort-pool check.

- **Estimates are ranges, never single numbers.** Always emit a range + basis (e.g. "12–18h; 3 modules, one high-churn; 2 similar past tickets at 14h/16h"). Single-point estimates are explicitly forbidden (cone-of-uncertainty). Use three-point or Monte Carlo.

- **Copilot, not autopilot.** The system only *recommends*; the PMO decides. Every decision emits an auditable **evidence chain** (`TriajKararı` / case file): clauses checked, best match + similarity, impacted modules, reasoning, confidence, **model/prompt version**, and **index freshness stamp** — so any decision can be reconstructed later for a contractual dispute. PMO decisions feed back to calibrate future estimates and thresholds.

- **Index freshness matters.** Stale indexes give wrong impact analysis. Re-index incrementally per commit/merge (or nightly), re-index high-churn modules more often, and stamp every triage decision with the index date.

**Where things go:** new vendor adapter → `adapters/<name>.py` + one branch in the relevant `build_*` in `adapters/registry.py` (core never imports it); new code-graph engine → produce the `CodeIndex` schema (`adapters/code_index.py`) like `ast_code_index`/`joern_code_repo`/`graphify_code_repo` do; new domain field → `core/models.py`; decision-logic change → `engine/triage.py`; new persistence backend → already covered by the connection string (SQLAlchemy); UI tweak → `api/web.py` + `api/templates/`. **Both indexers (Joern, AST) emit the same `CodeIndex` JSON**; **persistence sits behind the `CaseFileRepository` port** (SQL or in-memory) so the API/HITL never bind to a concrete DB.

## Tech stack

Implemented: Python 3.12+ (managed by **uv**), **FastAPI** + uvicorn, **Pydantic v2** + pydantic-settings, **PyYAML**, **httpx** (LLM/tracker clients), **MCP SDK** (`mcp`, stdio server), **SQLAlchemy 2.0** (+ **psycopg** for Postgres), **Jinja2** + **HTMX** (server-rendered UI, no Node build), **python-docx** + **openpyxl** + **pypdf** (document upload parsing), **ruff** + **mypy** + **pytest**. Ports are `typing.Protocol`; adapters satisfy them structurally (no inheritance). Code graph via **live Joern** (CPG; `pysrc2cpg`) with a dependency-free **Python `ast`** alternative; churn via `git log`. Scope extraction is heuristic with a **two-provider LLM seam** behind one `LLMClient` port: **Anthropic Claude API** (`llm_anthropic.py`, official **`anthropic` SDK**, `claude-opus-4-8`; `ETKI_LLM_PROVIDER=anthropic` + `ANTHROPIC_API_KEY`) or **OpenAI-compatible** (`llm_openai.py`, **Ollama** `qwen2.5:3b`/`llama3.2:3b` at `localhost:11434/v1`; `ETKI_LLM_BASE_URL`). `registry.build_llm_client(settings)` picks the provider (config, not code) and drives both scope extraction and the agent. Effort estimation is deterministic three-point/PERT (no RNG). Persistence: **SQLite** (default/tests) or **Postgres via Docker** (live) behind the `CaseFileRepository` port. **Containerization** via Dockerfile + docker-compose (app + Postgres); the container uses the `ast` code engine so it needs no JVM.

Planned for later phases (not yet wired): **PostgreSQL** (case files / decision history), **tree-sitter** + CodeQL options, RAG + Turkish legal model (HukukBERT) scope extraction, Turkish NLP request-splitting (BERTurk/spaCy-tr). Principle: build the core (decision engine, baseline manager, connector layer); adopt the components.

## Development conventions (planned, but adopt from day one)

- **Vendor-agnostic from day 1.** Define the three port Protocols first; write an **in-memory fake adapter** for every port so the engine is testable without real systems.
- **Walking skeleton first.** Prove one thin end-to-end slice (one document → one ScopeItem → one code module → one decision → one approval) before deepening any layer.
- **Gate-based phases.** Each phase has a measurable exit criterion (e.g. "for a known past CR, retrieve the correct scope clause *and* the correct code regions"); do not advance past a failing gate.
- **The `eval/` harness is a CI gate.** It back-tests against labeled historical CRs: retrieval precision/recall (Gate 1), agreement with historical PMO decisions ≥70–80% + out-of-scope precision/recall (Gate 2), and estimate calibration (does actual effort fall in the predicted range). A change that lowers `eval/` metrics is not "done".
- **Audit logging from Phase 0.** Structured logging + prompt/model version tagging is built in from the start (security/KVKK is expensive to retrofit), not added later.
